# 产品设计：`writing-skills` 技能工程化质量系统

## 1. 一句话定义

`writing-skills` 是给创建、维护、优化 Claude Code skill 的人使用的流程型技能，用来在新建或修改 skill 前后，系统判断触发描述、硬边界、必须步骤、验证路径是否足够可靠，减少 Agent 跳过规划路径、错误省略流程和反复人工介入。

## 2. 问题陈述

当前创建或优化 skill 时，失败不一定表现为“skill 完全不可用”，而是更隐蔽：

- Agent 触发了相关能力，但认为任务“过于简单”，跳过必须规划路径。
- Agent 没有按 skill 定义执行审查、诊断、验证等关键步骤。
- skill 使用过程中，除明确规则要求外，仍然频繁需要人工介入。
- 旧 skill 出现失败后，用户很难判断问题来自 skill 不适配、用户输入不足、Agent 执行偏差，还是任务本身超出边界。
- 修改 skill 时容易直接加规则、加文字、加限制，但没有判断这些修改是否真正降低了跳步和误用风险。

真实后果是：Agent 多次失败重试，用户被迫介入流程，最后还要人工检查 skill 结果是否可靠。

## 3. 需求证据

已知证据：

- 多次发生 Agent 跳过技能规划路径。
- Agent 会把本应执行完整流程的任务误判为简单任务。
- 结果导致人工介入 Agent 流程、Agent 多次失败重试、人工检查 skill 结果。
- 修改旧 skill 时，用户希望能提供失败经历，由 `writing-skills` 判断是否属于 skill 使用不适配问题，再决定是否修改。

仍未验证：

- 失败发生频率的具体数量。
- 每次失败平均浪费时间。
- 哪些 skill 最容易出现该问题。
- 修改后的 skill 是否能稳定降低人工介入次数。

## 4. 目标用户

第一目标用户是维护 `my-skills` 仓库、创建和优化 Claude Code skill 的人。

具体场景：

- 创建新 skill 时，需要确保它不是“看起来完整”，而是 Agent 实际调用后不会跳过关键流程。
- 优化旧 skill 时，用户已经观察到失败经历，例如跳过规划、误判简单、流程被省略、反复重试。
- 发布或使用 skill 前，需要审查它是否具备强触发描述、硬边界、必须步骤、验证标准和失败诊断路径。

动机：

- 降低 skill 运行中的人工介入。
- 降低 Agent 多次失败重试。
- 让 skill 的创建、修改、验收从“凭经验”变成“可检查流程”。

当前约束：

- 不能假设所有失败都能靠改 skill 解决。
- 不能让 `writing-skills` 变成泛泛的 prompt 润色工具。
- 不能让流程太复杂，以至于 Agent 又跳过流程本身。
- 修改 skill 前必须先诊断是否真是 skill 不适配。

## 5. 当前替代方案

目前替代方案主要是：

- 人工介入 Agent 流程，提醒它不要跳过步骤。
- Agent 失败后多次重试。
- 人工检查 skill 输出是否符合预期。
- 靠经验修改 skill 描述、边界或步骤。
- 事后根据失败结果补充更多规则。

真实代价：

- 失败定位慢：不知道问题是触发、边界、流程、验证，还是任务输入。
- 修改风险高：可能把一次执行偏差误当成 skill 设计缺陷。
- 质量不可复用：每次靠人工经验处理，难以沉淀成稳定方法。
- Agent 仍可能跳步：仅增加文字不一定能强制执行关键路径。

## 6. 最小切口

虽然选择的是理想版，但第一阶段的最小切口应聚焦在一个闭环：

> 输入一个新 skill 或旧 skill 失败经历，`writing-skills` 必须先诊断适配性问题，再给出修改方案，最后审查修改后是否降低跳步、误触发和人工介入风险。

这个切口足够验证价值，因为它覆盖了最关键风险：

- 创建时是否能预防跳过规划路径。
- 修改时是否能基于失败经历诊断问题。
- 修改后是否能审查流程可靠性。
- 是否能避免无根据地过度修改 skill。

## 7. 已确认前提

1. `writing-skills` 的核心价值不是“帮人写得更漂亮”，而是减少 Agent 在 skill 流程中误判“太简单所以跳过必须步骤”。  
   如果这个不成立，技能会偏向文案润色，而不是解决真实失败模式。

2. 第一版必须同时覆盖两类任务：创建新 skill，以及优化已有但不适配的 skill。  
   如果这个不成立，可以先砍掉一半范围，做成更窄的诊断或创建助手。

3. “正确执行流程”的判断标准应包括：不跳过规划路径、除明确规则要求外不随意人工介入、按 skill 定义完成验证或审查步骤。  
   如果这个不成立，后续很难判断 skill 是否真的变好，只能靠主观感觉。

4. 优化旧 skill 时，用户会提供失败经历，例如 Agent 怎么跳过步骤、哪里误判、哪里多次重试。  
   如果这个不成立，技能只能做通用建议，无法判断到底是 skill 不适配，还是 Agent 执行/用户输入/任务边界的问题。

5. `writing-skills` 需要先判断“是否真的是技能不适配问题”，再决定是否修改 skill。  
   如果这个不成立，它可能会过度修改 skill，把一次执行偏差误当成技能设计缺陷。

## 8. 三个方案对比

### 方案 A：最小验证版

做成一个 skill 质量审查与修补清单，重点检查触发描述、硬边界、必须步骤、禁止跳过条件。优点是验证快，风险是偏审查，不够覆盖完整创建和维护流程。

### 方案 B：平衡版

做成“创建 + 优化 + 适配性诊断”的流程型 skill。它能覆盖创建新 skill 和失败后优化旧 skill，要求先收集失败经历、诊断问题类型、再提出修改。优点是贴近当前痛点，风险是工程化沉淀不足。

### 方案 C：理想版

做成完整的 skill 工程化质量系统，覆盖设计、编写、审查、回归验证、失败复盘。它不只是改一个 skill，而是建立一套可复用的 skill 质量方法。优点是长期价值最大，风险是第一版容易过重，必须把流程拆成强制阶段，避免 Agent 自己跳过 `writing-skills` 的流程。

## 9. 推荐方案

用户选择：方案 C：理想版。

选择理由：

- 当前痛点已经不是单次 skill 编写，而是 skill 创建、优化、失败诊断、流程可靠性验证缺少系统方法。
- 反复出现的失败模式是流程级问题：Agent 跳过规划、误判简单、人工介入、多次重试。
- 方案 C 可以把这些问题沉淀成工程化流程，而不是每次靠人工提醒。
- 但实现时必须采用分阶段落地，第一阶段先完成“诊断—修改—审查—验证”的最小闭环。

## 10. 第一版范围

### 必须做

- 明确 `writing-skills` 的触发场景：
  - 创建新 skill。
  - 修改现有 skill。
  - 审查 skill 是否会被 Agent 跳过关键流程。
  - 根据失败经历判断 skill 是否不适配。
  - 验证 skill 修改后是否降低跳步和人工介入风险。

- 建立创建新 skill 的强制流程：
  - 先问目标使用场景。
  - 再定义目标用户和触发语。
  - 再定义硬边界。
  - 再定义必须步骤。
  - 再定义禁止跳过条件。
  - 最后定义完成标准和验证方法。

- 建立旧 skill 失败诊断流程：
  - 收集失败经历。
  - 判断失败发生在哪一层：触发、边界、流程、验证、用户输入、Agent 执行偏差。
  - 判断是否属于 skill 不适配。
  - 如果不是 skill 不适配，明确不修改或只给使用建议。
  - 如果是 skill 不适配，再提出最小修改方案。

- 建立修改前审查：
  - 为什么要改。
  - 改哪一类问题。
  - 哪些内容不应该改。
  - 是否可能增加误触发或流程负担。

- 建立修改后审查：
  - 是否更清楚地阻止 Agent 跳过规划路径。
  - 是否明确哪些步骤必须执行。
  - 是否明确哪些情况需要人工介入。
  - 是否避免把所有情况都推给用户确认。
  - 是否保留原 skill 意图。

- 建立验证用例：
  - 新建 skill 验证用例。
  - 旧 skill 失败复现用例。
  - 跳过规划路径的反例用例。
  - 不应触发 skill 的负例用例。
  - 修改后回归检查。

- 建立失败复盘模板：
  - 用户原始请求是什么。
  - Agent 实际做了什么。
  - 哪个必须步骤被跳过。
  - 是否发生人工介入。
  - 是否多次失败重试。
  - 初步判断是哪类问题。
  - 是否需要修改 skill。

### 明确不做

- 不一开始做自动化 benchmark 平台。
- 不承诺所有 Agent 执行失败都能靠修改 skill 解决。
- 不把所有 skill 强行改成同一种模板。
- 不替代 `/skill-creator` 的全部能力，而是补强 skill 工程化质量、适配性诊断和流程可靠性。
- 不在缺少失败经历时擅自大改旧 skill。
- 不把“用户觉得不顺手”直接等同于“skill 不适配”。

## 11. 验收标准

第一版是否值得继续，用以下可观察标准判断：

1. 创建新 skill 后，Agent 不再跳过规划路径。  
   验证方式：用一个新 skill 创建任务测试，观察是否先完成场景、触发、边界、流程、验证定义，而不是直接写文件。

2. skill 使用中，除明确规则要求人工介入外，Agent 能正确按流程执行。  
   验证方式：用真实任务触发 skill，检查是否无故要求用户确认、无故中断、无故跳过必须步骤。

3. 修改旧 skill 时，用户提供失败经历后，`writing-skills` 能先判断是否属于 skill 使用不适配问题。  
   验证方式：给出至少两个失败案例，一个属于 skill 不适配，一个不属于；检查它是否区分处理。

4. 修改 skill 前，必须输出诊断结论和修改理由。  
   验证方式：不允许直接进入编辑；必须先说明失败类型、证据、修改目标。

5. 修改 skill 后，必须输出适配性审核。  
   验证方式：检查是否说明修改如何降低跳步、误触发、人工介入或流程失败风险。

6. 连续 2-3 个真实 skill 案例中，人工介入次数明显下降。  
   验证方式：记录修改前后是否仍需用户提醒 Agent 执行规划、验证、诊断步骤。

## 12. 风险和反证

以下情况说明方向可能错了：

- Agent 仍然频繁跳过 `writing-skills` 自己定义的诊断、审查、验证流程。
- 用户提供失败经历后，它仍然直接改 skill，而不是先判断是否属于 skill 不适配。
- 它把所有失败都归因于 skill 文档不够详细，导致 skill 越改越长、越改越重。
- 修改后触发更泛，导致不该触发时也触发。
- 修改后流程过度依赖人工确认，反而增加人工介入。
- 新 skill 创建质量没有明显提升，仍需要用户反复提醒“先规划、再验证”。
- 它只产出漂亮的 skill 文档，但无法减少 Agent 多次失败重试。

## 13. 下一步行动

下一步不应直接大规模改所有 skill，而是做一次小范围真实验证：

1. 选择一个要新建的 skill，使用 `writing-skills` 的理想版流程创建。
2. 选择一个已经发生过“Agent 跳过规划路径”的旧 skill，提供失败经历，让 `writing-skills` 诊断是否是 skill 不适配。
3. 修改前记录失败点：跳过了什么、人工介入了几次、是否多次重试。
4. 修改后用同类请求回归测试。
5. 对比是否满足三个结果：
   - Agent 不再跳过规划路径。
   - 非规则要求下不再随意人工介入。
   - 修改前能判断是否真是 skill 不适配。
